# 提案：コマンド実行のログ出力追加

## 目的

すべての外部コマンド実行（`tokio::process::Command`, `std::process::Command`）をログに記録し、デバッグやトラブルシューティングを容易にする。

## 背景

現在、orchestratorは以下のような多くの外部コマンドを実行している：

- OpenSpec CLI (`openspec list`, `openspec apply` など)
- OpenCode / Claude Code などのAIエージェントコマンド
- VCSコマンド（`git`）
- ライフサイクルフック（ユーザー定義シェルコマンド）

しかし、現状では一部のコマンド（特にagent.rsでのapply/archive/analyzeコマンド）しかログに記録されておらず、VCSコマンドやその他の重要な実行はログに残らない。

これにより以下の問題が発生している：

- 並列実行時のVCSコマンド失敗の原因特定が困難
- フックが実行されたかどうかの確認が難しい
- システム全体の動作フローの把握が困難

## 提案内容

すべてのコマンド実行箇所で、実行前に以下の情報をログに出力する：

- コマンドライン全体（実行可能ファイル + 引数）
- 作業ディレクトリ（current_dirが設定されている場合）
- 実行コンテキスト（どのモジュールから実行されたか）

ログレベルは以下のように分類：

- `debug!`: VCSコマンド、内部的な補助コマンド
- `info!`: ユーザー向けの主要コマンド（apply, archive, analyze, hooks）

## 影響範囲

### 変更が必要なファイル

- `src/agent.rs` - 一部は対応済み、追加で `run_apply()` など非streaming版
- `src/parallel/executor.rs` - VCSコマンド全般
- `src/parallel/mod.rs` - workspace初期化コマンド
- `src/hooks.rs` - 一部対応済み、追加でWindows版コマンド
- `src/vcs/git/mod.rs` - すべてのgitコマンド
- `src/vcs/git/commands.rs` - git補助コマンド
- `src/parallel/cleanup.rs` - cleanup時のVCSコマンド

### 既存のログ出力との整合性

現在、`agent.rs`では既に以下の形式でログ出力している：

```rust
info!("Running apply command: {}", command);
```

この形式を他のコマンド実行箇所でも統一的に採用する。

### パフォーマンス影響

- ログ出力自体のオーバーヘッドは無視できるレベル
- デフォルトでは `info!` レベル以上のみ表示され、VCSコマンドのような詳細ログは `RUST_LOG=debug` で有効化

## 実装の非目的

- コマンド実行結果（stdout/stderr）の完全なログ記録は含まない（必要に応じて別機能として実装）
- コマンド実行時間の計測は含まない（既存のhistory機能で一部対応済み）
- 機密情報（APIキー、パスワード）のマスキングは含まない（現状でも環境変数で管理）

## リスク

- ログ量が増加し、重要な情報が埋もれる可能性
  - 対策：適切なログレベル分けとフィルタリングで対応
- コマンドライン引数に機密情報が含まれる可能性
  - 対策：現状でも環境変数経由で渡すのがベストプラクティス

## 承認基準

- [ ] すべてのCommand::new()呼び出しでログ出力が追加されている
- [ ] ログレベルが適切に設定されている（user-facing: info, internal: debug）
- [ ] 既存のテストがすべて通過する
- [ ] ログ出力フォーマットが統一されている
