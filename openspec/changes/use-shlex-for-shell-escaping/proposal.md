# Proposal: shlex によるサブコマンド全体のエスケープ強化

## 背景

現在のプレースホルダー展開（`{prompt}`, `{proposal}`, `{change_id}`, `{conflict_files}`）は手動エスケープに依存しており、シングルクォート以外の特殊文字や改行への対応が不十分です。これにより、以下の課題があります。

- コマンドインジェクションのリスク
- 文字列の複雑度に依存した不安定な動作
- サブコマンドごとにエスケープの一貫性がない

## 提案内容

信頼性の高い `shlex` クレートを用いて、サブコマンド全体のシェル引数エスケープを統一・確実化します。

### 変更対象

- `apply_command`, `archive_command`, `analyze_command`, `propose_command`
- hooks のコマンドテンプレート
- `{prompt}`, `{proposal}`, `{conflict_files}` などの埋め込みプレースホルダー

### 主要な改善点

1. **統一エスケープ**: `expand_prompt()`/`expand_proposal()`/`expand_conflict_files()` を `shlex::try_quote()` ベースに統一
2. **テンプレート互換性**: 既存テンプレートで `{prompt}` がクォートされている場合でも、二重クォートにならないよう互換レイヤーを用意
3. **クロスプラットフォーム配慮**: POSIX シェル向けに `shlex` を使用し、Windows (`cmd /C`) では既存の挙動を維持しつつ安全な文字列化を検討
4. **テスト強化**: 文字列境界・特殊文字・改行・マルチバイト文字に対するエスケープ検証を追加

## 期待される効果

- コマンドインジェクションの防止
- プレースホルダー展開の一貫性向上
- 特殊文字を含む入力の安全な取り扱い
- パラレル実行やフック実行での安定性改善

## 互換性

- 既存の設定ファイルは動作継続
- クォート済みテンプレート (`"claude '{prompt}'"`) でも壊れない互換挙動を提供
- 将来的には「クォート不要テンプレート」への移行を推奨

## 追加検討事項

- Windows でのエスケープ方式（`cmd` の引用符規則）をどこまで統一するか
- `{change_id}` など構造的に安全な値のエスケープ方針
