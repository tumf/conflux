## Context
現状のエラーは固定文字列が多く、運用時に対象変更や実行環境を特定しにくい。特に apply/archive/parallel 実行や AI コマンド実行の失敗時に、ログと TUI 出力の情報不足が調査コストを上げている。

## Goals
- エラー文言に実行文脈（操作種別・change_id・workspace/cwd・失敗理由）を必ず含める
- TUI とログのエラーメッセージを同一内容に揃える
- 既存のログレベルや出力形式のルールを壊さない

## Non-Goals
- エラーハンドリングの制御フロー変更
- リトライ回数や実行タイミングの変更
- 既存仕様の根本的な再設計

## Decisions
- 既存の OrchestratorError を使い、メッセージ本文のみを改善する
- 例外・キャンセル・内部失敗を区別し、文脈情報を付与する
- TUI/Parallel のイベントに流すエラー文字列は OrchestratorError と同一の文言にする

## Risks / Trade-offs
- 文字列が長くなり、ログ表示の可読性が下がる可能性がある
  - 対策: 既存の形式（単一行）を維持し、必須情報のみ付与する
