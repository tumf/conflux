# Proposal: テストカバレッジ分析の追加

## 背景

プロジェクトには既に69個のテストが存在し、2つの仕様（cli、configuration）が定義されている。しかし、仕様・テスト・実装の整合性が不明確な状態にある。

カバレッジ測定により以下の問題が明らかになった：

### 現状のカバレッジ

| モジュール | カバレッジ | 状態 | 未テスト行数 |
|-----------|-----------|------|--------------|
| hooks.rs | 86.35% | ✅ 良好 | 55行 |
| config.rs | 83.33% | ✅ 良好 | 42行 |
| agent.rs | 73.81% | ⚠️ まあまあ | 66行 |
| openspec.rs | 73.08% | ⚠️ まあまあ | 35行 |
| progress.rs | 60.29% | ❌ 低い | 27行 |
| tui.rs | 37.25% | ❌ 低い | 758行 |
| orchestrator.rs | 28.14% | ❌❌ 非常に低い | 286行 |
| opencode.rs | 8.82% | ❌❌ 非常に低い | 93行 |
| **全体** | **50.47%** | - | **1424行** |

### 特定された問題

1. **仕様に記載されているがテストされていない振る舞い**
   - 環境変数 `OPENSPEC_CMD` の優先順位（CLI引数 > 環境変数 > デフォルト）
   - プロジェクト設定とグローバル設定の優先順位
   - TUIの自動更新機能（5秒間隔）
   - 最小ターミナルサイズの処理

2. **実装されているが仕様化されていない可能性がある振る舞い**
   - `opencode.rs`: 8.82%カバレッジ → 91.18%の実装が未テスト
   - `orchestrator.rs`: 28.14%カバレッジ → 71.86%の実装が未テスト
   - これらの未テストコードが仕様化されているか不明

3. **テストツールと継続的な分析プロセスの不在**
   - カバレッジ測定の標準化されたコマンドがない
   - 仕様とテストのマッピングドキュメントがない
   - カバレッジレポートの継続的な監視プロセスがない

## 目的

**カバレッジを品質目標ではなく、診断ツールとして活用する。**

以下を実現する：

1. **仕様・テスト・実装の整合性を確保**
   - 仕様に記載されている振る舞いが全てテストされていることを検証
   - テストされていない実装が仕様化されているか確認

2. **継続的な分析プロセスの確立**
   - カバレッジ測定の標準化
   - 仕様とテストのマッピングドキュメント作成
   - ギャップ検出プロセスの確立

3. **不足しているテストの追加**
   - 仕様に基づく優先度付き
   - 特に低カバレッジモジュール（opencode.rs, orchestrator.rs）の分析

4. **未仕様化された振る舞いの文書化**
   - 実装されているが仕様化されていないロジックの特定
   - 必要に応じて仕様への追加

## スコープ

### 含まれるもの

- カバレッジ測定コマンドとドキュメント
- 仕様とテストのマッピングドキュメント
- 不足しているテストの追加（仕様ベース）
- カバレッジ分析プロセスの確立
- 未仕様化された振る舞いの特定と文書化

### 含まれないもの

- カバレッジ率の目標設定（カバレッジは診断ツール）
- CI/CD統合（別の変更で対応）
- パフォーマンステスト

## 成功基準

1. 全ての仕様シナリオに対応するテストが存在する
2. カバレッジレポートを生成する標準化されたコマンドが文書化されている
3. 仕様とテストのマッピングドキュメントが作成されている
4. 低カバレッジモジュール（<50%）の未テストコードが分析され、適切に対応されている
5. 未仕様化された振る舞いが文書化または削除されている

## 影響範囲

- テストコード（`src/**/*.rs` の `#[cfg(test)]` モジュール）
- ドキュメント（新規作成）
- 仕様（必要に応じて追加）
- ビルド設定（なし - `cargo-llvm-cov` は既にインストール済み）

## リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| 大量のテスト追加による実装遅延 | 中 | 優先度付け：仕様シナリオのみ対象 |
| 未仕様化された振る舞いが多数発見される | 高 | 段階的対応：まず文書化、次に削除または仕様化 |
| カバレッジ測定の実行時間増加 | 低 | 並列実行とキャッシュ活用 |
