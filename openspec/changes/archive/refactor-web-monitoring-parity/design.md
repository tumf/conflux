## Context
Web UI は WebState の changes スナップショットのみを購読しており、TUI の画面状態（キュー、ログ、ワークツリー、実行中状態など）と一致しない。更新経路も TUI の ChangesRefreshed の一部に限定され、TUI と同じ情報をリアルタイムに再現できない。

## Goals / Non-Goals
- Goals:
  - Web UI が TUI と同じ情報を同じ更新タイミングで受け取れる
  - 監視状態の単一ソースを定義し、TUI と Web で共有する
  - 既存の Web 監視機能と互換性を維持する
- Non-Goals:
  - Web UI のデザイン刷新のみを目的とした変更
  - 新しい外部サービスの導入

## Decisions
- Decision: TUI の内部状態と Web 監視の状態を統合する共通ステートモデルを導入する
  - TUI が消費するイベントを同じモデルに反映し、Web はそのモデルを購読する
  - 状態配信のチャネルは WebSocket を主軸とし、REST はスナップショット取得のために維持する
- Decision: Web で必要な状態は段階的に配信し、最初に「完全一致」の状態モデルを定義する
  - 変更一覧、キュー状態、ログ、ワークツリー、実行中ステータスを含む

## Risks / Trade-offs
- State の統合により TUI の責務が増え、イベントフローが複雑化する
  - Mitigation: 状態更新の責務を専用モジュールに移し、TUI からはイベントを送るだけにする
- Web への配信データが増えることで帯域・描画コストが増加する
  - Mitigation: 差分配信または段階的サブスクライブの検討をタスク化する

## Migration Plan
1. 共通ステートモデルとイベント更新 API を追加
2. TUI のイベント更新を共通モデルへ移行
3. Web UI の購読先を共通モデルに切り替え
4. 互換性確認と旧 WebState の廃止計画を策定

## Open Questions

- Web UI は TUI と同じレイアウト/情報量を必須とするか、表示の最適化を許容するか
