# Design: テストカバレッジ分析システム

## アーキテクチャ

### カバレッジ測定ツール

**選定**: `cargo-llvm-cov`

**理由**:
- macOS (Darwin) で動作 ✅
- 既にインストール済み ✅
- LLVM ベースの高精度カバレッジ
- HTML/LCOV/JSON 形式のレポート生成

### カバレッジの役割

**カバレッジは診断ツールであり、品質目標ではない。**

```
┌─────────────┐     ┌──────────┐     ┌────────────┐
│   仕様      │────►│ テスト   │────►│  実装      │
│  (Specs)    │     │ (Tests)  │     │  (Code)    │
└─────────────┘     └──────────┘     └────────────┘
       │                  │                  │
       └──────────────────┴──────────────────┘
                      │
                      ▼
              ┌─────────────┐
              │ カバレッジ  │ ← 診断ツール
              │   測定      │
              └─────────────┘
                      │
                      ▼
        ┌─────────────────────────────┐
        │ ギャップ検出                │
        │ 1. 仕様あり → テストなし    │
        │ 2. 実装あり → 仕様なし      │
        └─────────────────────────────┘
```

## カバレッジ分析プロセス

### Step 1: カバレッジ測定

```bash
# HTMLレポート生成
cargo llvm-cov --all-features --workspace --html

# サマリー表示
cargo llvm-cov --all-features --workspace --summary-only

# LCOV形式（CI用）
cargo llvm-cov --all-features --workspace --lcov --output-path target/coverage.lcov
```

### Step 2: 仕様とテストのマッピング

**マッピングドキュメント構造**:

```markdown
# 仕様 → テスト マッピング

## Configuration仕様

### Requirement: Environment Variable Configuration
- ✅ Scenario: 環境変数のみ設定
  - Test: `test_env_var_openspec_cmd`
  - File: `src/config.rs:XX`
- ❌ Scenario: CLI引数が環境変数より優先
  - Status: テストなし
  - Action: テスト追加が必要

### Requirement: エージェントコマンドの設定ファイル
- ✅ Scenario: プロジェクト設定ファイルが存在する場合
  - Test: `test_get_commands_with_custom_values`
  - File: `src/config.rs:XX`
...
```

### Step 3: ギャップ分析

**2つのギャップタイプ**:

1. **仕様ギャップ**: 仕様に記載されているがテストされていない
   - **検出**: 仕様シナリオリスト vs テストリスト
   - **対応**: テスト追加

2. **実装ギャップ**: 実装されているが仕様化されていない
   - **検出**: カバレッジ < 70% のモジュールの未テストコード分析
   - **対応**: 仕様化 or 削除

### Step 4: 優先度付け

| 優先度 | 条件 | 対応 |
|--------|------|------|
| P0 | 仕様シナリオに対応するテストがない | 即座にテスト追加 |
| P1 | カバレッジ < 30% のコアモジュール | 分析して仕様化 or テスト追加 |
| P2 | カバレッジ 30-70% のモジュール | 段階的にテスト追加 |
| P3 | カバレッジ > 70% のモジュール | 現状維持、新機能追加時に改善 |

## モジュール別戦略

### opencode.rs (8.82%)

**現状**:
- 93行が未テスト
- わずか1個のテスト (`test_opencode_runner_creation`)

**分析アプローチ**:
1. 実装されているメソッドをリスト化
2. 各メソッドが仕様に記載されているか確認
3. 仕様化されていないメソッドの用途を特定
4. 仕様化 or 削除を判断

**テスト戦略**:
- `run_command_streaming`: ストリーミング出力のテスト
- `run_command`: 基本コマンド実行のテスト
- `analyze_dependencies`: 依存関係分析のテスト
- エラーハンドリングのテスト

### orchestrator.rs (28.14%)

**現状**:
- 286行が未テスト
- 5個のテスト（フィルタリングロジックのみ）

**分析アプローチ**:
1. CLI仕様のオーケストレーション関連要件を確認
2. 実装されているが未テストのメソッドを特定
3. 仕様に記載されているが未テストのシナリオを特定

**テスト戦略**:
- `select_next_change`: 変更選択ロジック
- `apply_change`: 変更適用プロセス
- `archive_change`: アーカイブプロセス
- `analyze_with_llm`: LLM分析（モック化）
- エラーハンドリングとリトライロジック

### tui.rs (37.25%)

**現状**:
- 758行が未テスト
- 20個のテスト（状態管理のみ）

**分析アプローチ**:
1. CLI仕様のTUI関連要件を確認
2. レンダリングロジックとイベントハンドリングの分離
3. テスト可能な状態管理とテスト困難なレンダリングの区別

**テスト戦略**:
- 状態管理ロジック: 既存テストを拡張
- 自動更新ロジック: タイマーモックでテスト
- レイアウト計算: 最小限のテスト（複雑度高）
- レンダリング: 手動テスト推奨（ratatui依存）

## 継続的分析プロセス

### 新機能追加時

```
1. 仕様にシナリオ追加
2. シナリオに対応するテスト作成
3. 実装
4. カバレッジ測定
5. ギャップ確認
```

### 定期レビュー（月次）

```
1. カバレッジレポート生成
2. マッピングドキュメント更新
3. 新しいギャップの特定
4. 優先度付けと対応計画
```

## テスト設計原則

### 仕様ベーステスト

**原則**: 全ての仕様シナリオに対応するテストが存在する

```rust
// ❌ 悪い例: 実装を直接テスト
#[test]
fn test_internal_parsing_logic() {
    // 仕様に記載されていない内部ロジックのテスト
}

// ✅ 良い例: 仕様シナリオをテスト
#[test]
fn test_env_var_overrides_default_openspec_cmd() {
    // 仕様: "Scenario: 環境変数のみ設定"
    // WHEN: 環境変数 OPENSPEC_CMD が設定されている
    // THEN: その値が使用される
}
```

### テストの粒度

| レベル | 対象 | ツール |
|--------|------|--------|
| ユニット | モジュール単位の振る舞い | `#[cfg(test)]` モジュール |
| 統合 | モジュール間の連携 | `tests/*.rs` |
| E2E | エンドツーエンドのワークフロー | `tests/e2e_tests.rs` |

### モックとスタブ

```rust
// 外部依存のモック化
#[test]
fn test_opencode_execution_with_mock() {
    // opencode コマンドの代わりにモックスクリプトを使用
    let mock_script = create_mock_script();
    let runner = OpenCodeRunner::new(mock_script);
    // ...
}
```

## 成功指標

### 定量指標

- ❌ カバレッジ率 XX% 以上（目標ではない）
- ✅ 仕様シナリオのテストカバー率 100%
- ✅ カバレッジ < 30% のモジュール数 0個

### 定性指標

- ✅ 全ての実装が仕様に記載されている or テストで検証されている
- ✅ カバレッジレポートからギャップを特定できる
- ✅ 新機能追加時に仕様・テスト・実装の整合性を維持できる
