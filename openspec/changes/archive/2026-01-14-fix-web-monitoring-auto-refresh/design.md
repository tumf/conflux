## Context
Webダッシュボードはオーケストレーター状態の可視化を目的としますが、現状ではステータスが更新されず、手動リロードでも最新にならないケースが報告されています。

この症状は、主に以下のどれか（または複合）で起きる可能性があります。
- WebSocket が接続できていない／更新がブロードキャストされない
- 接続できているが、クライアント側で描画更新が反映されていない
- `GET /api/state` 等のレスポンスがブラウザ/中間層キャッシュで stale になっている
- 初期ロード時に“最新スナップショット”を取得せず、更新イベント待ちになっている

## Goals / Non-Goals
- Goals:
  - ページを開いた直後から、最新の状態が表示される
  - オーケストレーター状態が変化したら、ページリロード無しで追随する
  - WebSocket が使えない/不安定でも、一定の表示更新が保証される
  - 手動リロードで stale 状態を表示しない
- Non-Goals:
  - UI の全面刷新
  - 新しい監視 API の追加

## Decisions
- Decision: WebSocket を主経路、ポーリングをフォールバック
  - Why: WebSocket は低レイテンシな更新に適し、フォールバックのポーリングで信頼性を担保できる
- Decision: フォールバック・ポーリング間隔はデフォルト 5s
  - Why: 過剰なリクエストを避けつつ、監視用途として十分な追随性を確保する
- Decision: `GET /api/state` にキャッシュ抑止ヘッダ（`Cache-Control: no-store`）を必須化
  - Why: “手動リロードでも更新されない”症状に対して、ブラウザキャッシュの影響を最小化できる
- Decision: WebSocket 接続時に初期スナップショットを送信
  - Why: 更新イベントが発生しない期間でも、初回表示が古いまま残るのを防ぐ

## Risks / Trade-offs
- ポーリングは負荷増の可能性がある → WebSocket 優先、間隔を控えめにし、切断時のみ有効化する
- キャッシュ抑止により一部環境で帯域/リクエストが増える → 状態系エンドポイントに限定する

## Migration Plan
- 実装は互換性維持のまま段階導入（ヘッダ追加 → 初期スナップショット → フォールバックポーリング）
- 既存設定があれば、それを上書きしない形でデフォルトを提供する

## Open Questions
- “最終更新時刻”の表示は必須か（デバッグ性 vs 画面の簡潔さ）
- WebSocket 断の判定猶予（即時切替 vs 数秒待ってから切替）
