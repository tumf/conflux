# Design: `+` で worktree を作り `worktree_command` を実行

## ゴール

- TUIの `+` を「提案入力UI」ではなく「提案作成フローの起点」にする
- 提案作成は隔離された Git worktree 上で行えるようにする
- 設定が無い/条件が満たせない場合は無操作（UX的に邪魔をしない）

## 非ゴール

- 提案作成コマンドの中身（`worktree_command` の具体的な実装内容）を規定しない
- 作成済み worktree の自動削除は行わない（本変更では残す）

## 基本フロー

1. ユーザーがSelectモードで `+` を押す
2. `worktree_command` が設定されているか確認する
3. 現在の作業ディレクトリが Git リポジトリ上か確認する（検出できない場合は無操作）
4. 一時ディレクトリ配下に新しい Git worktree を作成する
   - ベースは現在の `HEAD` とする
   - worktreeは **削除しない**
5. 作成した worktree を **子プロセスの `cwd`** として `worktree_command` を実行する

## 一時ディレクトリ

- 既存の parallel 実行で利用している「workspaces のベースディレクトリ」の考え方と整合させる
- 具体的な作成先は以下のどちらかとする
  - `workspace_base_dir` が設定されていればそれを使用する
  - 未設定なら OS の system temp directory を使用する

## プレースホルダー

`worktree_command` はテンプレートとして扱い、以下のプレースホルダーを展開できるようにする。

- `{workspace_dir}`: 作成した worktree の絶対パス（実行時の `cwd` と一致）
- `{repo_root}`: 元リポジトリのルート

展開とシェルエスケープの規則は、既存の `{proposal}` / `{prompt}` 等の展開と同等とする。

## 無操作の定義

以下の条件では **画面遷移・警告・ログ追加を行わず** に無操作とする。

- Gitリポジトリとして検出できない
- `worktree_command` が設定されていない

## 互換性と移行

- 既存の `propose_command` は設定として残り得るが、`+` キーの挙動としては利用しない
- 旧来の Proposing（テキスト入力UI）を廃止するため、関連状態・キーヒントは削除または無効化する
