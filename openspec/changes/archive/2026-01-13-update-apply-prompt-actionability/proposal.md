# 提案：apply プロンプトで「実行可能タスク」へ正規化し、スタックを防ぐ

## 目的

apply ループが「実装は成功しているのに tasks.md の抽象タスクが残る」ことで繰り返される問題を防ぎ、オーケストレーターの自動進行を安定させる。

## 背景

現状の apply では、エージェントが tasks.md のタスクを完了したとしても、最後に残る検証・確認タスクが抽象的だとチェックできず、進捗が 100% にならない。
その結果、apply 自体は成功（exit code 0）しているのに、オーケストレーターは「未完了タスクがある」と判断して apply を再実行し続け、スタックのように見える。

例（問題になりやすい表現）:
- 「エラー発生時のトラブルシューティングが容易になったことを確認」
- 「ログが期待通り出力されることを確認」

これらは“意図”としては妥当だが、
- どのコマンドを実行するのか
- どの出力・状態になれば合格なのか
が明確でないため、エージェントが完了判定できない。

## 提案内容

ハードコードされた apply システムプロンプト（`APPLY_SYSTEM_PROMPT`）を改善し、未完了タスクが必ず「実行可能（actionable）」であるようにエージェントへ強制する。

方針（ユーザー要望）:
- 抽象タスクは極力「具体コマンド + 合格基準」に書き換えて残す（Future work 送りは最後の手段）

具体的な変更:
- tasks.md の未完了チェックボックスは必ず「実行可能タスク」でなければならない
- 実行不能（抽象/主観/環境依存/人間判断必須）なタスクが見つかった場合
  - まず、複数の実行可能タスクへ分解・書き換える
  - それでも人間判断が必須なら `(future work)` を付けて `Future work` セクションへ移動し、チェックボックスから外す
- apply が成功（exit code 0）したのに未完了タスクが残る状態は“未解決”として扱い、tasks.md の正規化を優先して解消する

## 影響範囲

- 影響する仕様: `configuration`（apply_prompt とハードコードプロンプトの要件）
- 影響する実装箇所（予定）:
  - `src/agent.rs` の `APPLY_SYSTEM_PROMPT` 文面
  - `src/agent.rs` の関連テスト（プロンプト内容の期待値）
  - 必要に応じて設定テンプレート/例（文面の説明補強）

## 非目的

- tasks.md を自動解析して「実装の成否」をコード側で判定するロジックの追加
- apply ループ自体の制御ロジック（リトライ抑制など）の変更
- すべての検証を完全自動化すること（ただし“実行可能な検証タスク”へ落とすことは必須）

## リスク

- 過度に厳しいプロンプトにより、エージェントが tasks.md を大きく書き換えてしまう可能性
  - 対策: 「意図を保持して、検証可能な形に分解する」ことをルール化し、Future work は最後の手段とする
- タスクの書き換えが冗長になり、tasks.md が長くなる可能性
  - 対策: 1タスクを小さく、検証は最小のコマンドセットへ集約する（例：`cargo test`、`cargo clippy`、`cargo fmt --check`）

## 承認基準

- [ ] apply システムプロンプトに「未完了タスクは実行可能であること」および「抽象タスクの書き換え義務」が明記されている
- [ ] `configuration` の delta spec に、プロンプトに含まれるべき要件が追加されている
- [ ] `openspec validate update-apply-prompt-actionability --strict` が成功する
