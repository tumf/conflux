# 設計：apply プロンプトの「実行可能タスク」正規化

## Context

apply ループは `tasks.md` のチェック状態（完了/未完了）を進捗として扱う。
そのため、未完了チェックボックスが「実行/完了判定できない」内容だと、実装が完了していても永続的に未完了となり、apply が繰り返されてしまう。

## Goals / Non-Goals

### Goals
- 未完了タスクが常に“エージェントが実行可能な形”になっている
- 抽象タスクは可能な限り具体タスクに分解して保持する
- それでも実行不能な場合のみ `(future work)` としてチェックリストから外す

### Non-Goals
- オーケストレーター本体に「抽象度の判定」や「自動リライト」ロジックを追加する
- 人間の最終判断が必要な品質評価を完全自動化する

## Key Definitions

### 実行可能（Actionable）タスク
以下を満たすタスクを“実行可能”と定義する。
- このリポジトリ上で、エージェントが今すぐ完了できる
- 完了条件が客観的（合否が判断できる）

例:
- `cargo test` を実行し成功する
- `cargo clippy -- -D warnings` を実行し警告がない
- 特定のファイルに特定の文面を追加し、grep で確認できる

### 実行不能（Non-actionable）タスク
以下のようなタスクは“実行不能”とみなす。
- 主観評価: 「良い感じ」「使いやすい」「容易になった」
- 手動/目視だけで完了条件が不明確: 「確認する」
- 外部環境依存/人間判断必須で、具体的な代替検証が提示されていない

## Decision

### Decision: システムプロンプトで強制する
`apply_prompt` に加えて常に注入されるハードコードプロンプト（`APPLY_SYSTEM_PROMPT`）に、以下を明記する。
- 未完了チェックボックスは必ず実行可能であること
- 実行不能タスクを見つけたら、まず実行可能タスクへ分解・書き換えること
- どうしても人間判断が必須な場合に限り `(future work)` としてチェックボックスから外すこと
- apply 成功（exit 0）なのに未完了が残る状態は許容しないこと

理由:
- 設定ファイル `apply_prompt` が空でも確実に効く
- 実装変更は文面更新とテスト更新が中心で、最小差分で効果がある

## Rewrite Policy（A 方針）

- 原則: Future work に逃がさず、実行可能な検証へ落とす
- 典型的な置換:
  - 「容易になったことを確認」→
    - `RUST_LOG=debug ...` でログに必要情報が出ること
    - 失敗時にどのログが欠けるかを明記
  - 「期待通り出力されることを確認」→
    - 具体コマンド + 期待するログ/出力行（1〜3個）

## Risks / Trade-offs

- tasks.md の書き換え量が増える可能性
  - ただし、進捗の機械判定を安定させるために必要

## Migration Plan

- 既存 change の tasks.md は即時には変更しない
- 以降の apply 実行で“抽象タスクの書き換え”が促され、スタックを予防する

## Open Questions

- 「ファイル観察（特定文字列の存在）」を検証として許容する範囲をどこまで広げるか
  - 初期は「コマンド実行 + 合否」優先で運用し、必要に応じて拡張する
