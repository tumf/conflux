## Context
設定モジュールは単一ファイルに多責務が集約されており、設定キー追加や検証ルール変更時に副作用の把握が難しい。外部公開される挙動（設定優先順位、CLI互換性）は維持したまま、内部構造のみを整理する必要がある。

## Goals / Non-Goals
- Goals:
  - 責務単位の分割で可読性と変更容易性を改善する
  - 既存の設定解決結果と外部挙動を維持する
  - テスト容易性を高め、回帰を検知しやすくする
- Non-Goals:
  - 新規設定キーの追加
  - 設定優先順位ルールの変更
  - CLIオプションやJSONスキーマの仕様変更

## Design
- `mod.rs` は公開型と再エクスポートに限定する
- 読み込み優先順位ロジックを専用モジュールに分離する
- デフォルト値・検証ルール・変換処理を独立モジュール化する
- 既存の `serde` 互換性を保つため、フィールド名・デフォルト関数・シリアライズ挙動は変更しない

## Trade-offs
- 分割初期はファイル数増加で探索コストが上がるが、責務境界が明確になることで長期保守性を改善できる
- 内部関数の可視性調整が必要になるため、モジュール境界の設計を慎重に行う

## Verification
- 既存挙動を固定するキャラクタリゼーションテストを先に追加し、分割後も全件通過させる
- `cargo test`、`cargo fmt --check`、`cargo clippy -- -D warnings` で回帰と品質を確認する
