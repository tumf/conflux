## 背景
- 並列実行の order-based ループがディスパッチ処理を await するため、apply 実行中は再分析ループが停止する。
- その結果、apply 中に queued が増えても re-analysis が走らず、queued から apply への遷移が発生しない。

## 目標
- re-analysis ループを apply 実行と並行して継続できる構造にする。
- キュー変化やタイマーで再分析が起動し、空きスロットが出次第ディスパッチされる。

## 非目標
- 依存解決ルールや order 生成アルゴリズムの変更。
- worktree 作成/再利用/merge 手順の再設計。

## 方針
- ディスパッチを `JoinSet` などで非同期タスク化し、ループは in-flight を監視し続ける。
- in-flight の数から available_slots を算出し、空きが出たら即座に次をディスパッチする。
- queue の通知と join 完了イベントを同列のトリガとして扱う。

## トレードオフ
- in-flight 管理が増えるため、状態遷移の一貫性チェックが必要になる。
- 再分析頻度が上がるため、デバウンスとキャッシュ運用の調整が必要。

## 失敗時の扱い
- ディスパッチタスクの失敗は in-flight から外し、失敗理由を記録する。
- apply 中断時は cancel token で全 in-flight を停止させる。
